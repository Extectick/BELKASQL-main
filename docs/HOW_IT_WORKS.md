# Как устроен BELKASQL

Этот документ описывает не «идеальную архитектуру вообще», а именно то, как текущий репозиторий реально собран и где у него границы применимости.

Если нужен профиль, где клиенты ходят напрямую на оба DB-хоста, дополнительно смотрите:

- [INSTALL_DB_INGRESS.md](INSTALL_DB_INGRESS.md)
- [OPERATIONS_DB_INGRESS.md](OPERATIONS_DB_INGRESS.md)

## Базовая идея

Проект строится вокруг трёх узлов `etcd` и двух PostgreSQL-узлов под управлением Patroni:

- `city-a`
- `city-b`
- `cloud`

На DB-хостах живут:

- `etcd`
- PostgreSQL
- Patroni
- PgBouncer
- локальный HAProxy
- `pgBackRest`
- `postgres-exporter`
- `node-exporter`
- `promtail`

В облаке, в зависимости от профиля, живут:

- третий `etcd`
- облачный ingress на HAProxy + Keepalived
- MinIO для backup’ов
- Prometheus, Grafana, Loki, Alertmanager

## Роли по каталогам

### `db-node`

`db-node/docker-compose.yml` поднимает сразу несколько сервисов:

- `etcd`
- `db`
- `postgres-exporter`
- `node-exporter`
- `promtail`
- `local-lb`

Что это значит практически:

- каждый DB-хост содержит собственный локальный член `etcd`
- PostgreSQL и Patroni живут в одном контейнере `db`
- PgBouncer слушает внутри `db`-контейнера и публикуется наружу
- `local-lb` принимает SQL на `5000/5001`

Поведение `local-lb`:

- write-трафик идёт на локальный DB-узел, если тот primary
- если локальный узел replica, write может быть перекинут на удалённый DB-хост
- read-трафик идёт через отдельный HAProxy-listener

Это удобно, но важно понимать: доступность здесь зависит не только от Patroni, но и от того, что один DB-хост может достучаться до сервисных портов другого.

### `control-node`

`control-node` это не «центр управления», а довольно узкая роль:

- третий участник `etcd`
- `node-exporter`
- `promtail`

Он нужен для quorum в трёхузловой DCS-схеме и для наблюдаемости.

### `lb-node`

`lb-node` используется в классическом профиле и поднимает:

- HAProxy
- Keepalived
- `node-exporter`
- `promtail`

Назначение:

- принимать публичный клиентский трафик на `5000/5001`
- отправлять его на `city-a`/`city-b`
- при использовании двух LB-узлов держать VIP через Keepalived

Честная оговорка:

- локальная лаборатория не доказывает реальную корректность Keepalived/VIP между разными Linux-хостами

### `storage-node`

`storage-node/docker-compose.yml` поднимает:

- `minio`
- `node-exporter`
- `promtail`

Опционально с профилем `admin` поднимаются ещё:

- `minio-admin`
- `minio-mirror`

Назначение:

- `minio` хранит backup’ы `pgBackRest`
- `minio-admin` инициализирует replication rules
- `minio-mirror` запускает `mc mirror --watch` для непрерывного зеркалирования

Честная оговорка:

- если `storage-node` поднят без `--with-admin`, репликация во вторичный MinIO сама по себе не появится

### `observability-node`

`observability-node` поднимает:

- Prometheus
- Loki
- Alertmanager
- Grafana

Сбор логов и метрик:

- `promtail` на каждой роли читает Docker socket и отправляет логи в Loki
- Prometheus скрейпит `postgres-exporter`, `node-exporter`, `etcd`, HAProxy и MinIO
- конфигурация Prometheus генерируется на старте из шаблона и `env`

## Patroni и PostgreSQL

В проекте заложен такой режим:

- `synchronous_mode: true`
- `synchronous_mode_strict: false`
- `synchronous_node_count: 1`

Практический смысл:

- когда оба DB-узла живы, запись должна быть синхронной
- если один DB-узел пропадает, Patroni может ослабить требование и оставить запись доступной
- за эту доступность платят риском потери части последних подтверждённых транзакций в узком окне аварии

Это видно и в `test-failover.sh`: после исчезновения второго узла тест явно ожидает, что путь записи останется рабочим.

## `etcd` и кворум

Схема кворума здесь всегда трёхчленная:

- `etcd-city-a`
- `etcd-city-b`
- `etcd-cloud`

Для production-`env` bootstrap задаёт:

- `heartbeat-interval=2000`
- `election-timeout=10000`

Почему это важно:

- при трёх членах система нормально переживает один отказ из здорового состояния
- если после первого отказа не дождаться стабилизации и потерять ещё один критичный узел, автоматический failover уже не гарантирован

Отдельная тонкость DB-ingress профиля:

- локальный DB-узел должен ходить к своему colocated `etcd` через compose-имя `etcd-city-a` или `etcd-city-b`
- если заставить его ходить к самому себе через внешний/VPN IP, failover может зависнуть

## Резервные копии и DR

Источник истины для backup’ов:

- `pgBackRest` на DB-узлах
- основной S3-совместимый endpoint это `minio-primary`

Что происходит в реальности:

1. Скрипты пытаются сделать backup с replica.
2. Если в текущей схеме standby-local backup не проходит, они честно переключаются на primary.
3. Затем проверяется наличие данных и правил репликации во вторичном MinIO.

Это ограничение не скрыто в коде:

- `backup-status.sh` и `disaster-recovery.sh` оба содержат fallback на primary

Почему так:

- для «чистого» standby-backup часто нужна отдельная схема `pgBackRest` protocol/remote repo
- текущий стенд проще: backup-логика локальна для каждого DB-узла

Следствие:

- документы не должны обещать, что backup всегда идёт только с replica

## Переключение и возврат узла

Patroni выбирает primary через `etcd`.

Когда старый primary возвращается:

- включён `use_pg_rewind=true`
- ожидается автоматическое присоединение как replica

Но документация должна быть честной:

- `test-failover.sh` проверяет наличие `pg_rewind` в логах и сам пишет, что rejoin мог произойти и другим способом, например через timeline recovery

То есть корректнее говорить не «всегда будет `pg_rewind`», а «обычно ожидается `pg_rewind`, но возможен и другой механизм совместимого восстановления».

## Наблюдаемость

Что есть реально:

- базовые dashboard’ы Grafana
- статически описанные alerts для потери exporter’ов, MinIO и высокого давления по диску
- логирование контейнеров через `promtail` + Docker socket

Чего здесь нет:

- сложной маршрутизации алертов
- зрелой multi-tenant схемы логов
- богатой операционной модели для расследований

То есть observability есть и полезна, но это практический минимальный набор, а не полноценная enterprise-платформа.

## Что проект подтверждает, а что нет

### Подтверждается текущими скриптами и лабораторией

- `docker compose`-конфиги собираются с заданными `env`
- Patroni и `etcd` поднимаются и видят друг друга
- failover между двумя DB-узлами возможен
- backup и scratch restore из вторичного MinIO можно проверить скриптом
- Prometheus и Loki могут собирать метрики и логи со всех ролей

### Не подтверждается автоматически

- реальный DNS failover у вашего провайдера
- корректность маршрутизации между датацентрами
- корректная настройка firewall/ACL/VPN
- устойчивость к произвольной серии нескольких отказов подряд
- безопасность production-секретов только за счёт этого репозитория
